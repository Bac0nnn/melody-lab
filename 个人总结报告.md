## 一、项目概览

**Melody Lab** 是一个基于 Web Audio API 的交互式音乐创作与学习工具。项目包含 4 个八度全尺寸钢琴键盘（C2–B5）、和弦/音阶探索器、实时音频可视化（频谱/波形/圆形）、钢琴弹奏录制回放、系统音频转谱录制（自动音高识别转 MIDI）、以及预设钢琴曲播放等功能。代码以零依赖单文件 HTML（约 960 行）形式提供，同时保留了 Next.js 版本用于 Vercel 部署。Git 提交跨越 3 天，共 9 个 commit，完整覆盖项目开发全流程。

## 三、解决的关键技术问题

### 1. 钢琴键盘黑键定位算法

**问题：** 黑键需要精确"嵌入"白键之间，原定位公式使用 `BLACK_SEMIS = {1:0.7, 3:1.7, 6:3.7, 8:4.7, 10:5.7}`，导致 C# 被画在了 C 键内部而非 C-D 交界处，D#、F# 等也存在类似偏移。

**解决：** 分析了公式 `((oct-firstOctave)*7 + pos) * (100/7) - 1.5` 的数学含义后，发现 `pos` 值应代表"边界位置"（整数 1.0, 2.0, 4.0, 5.0, 6.0），修正后每个黑键正好卡在相邻白键的边界上，4 个八度 20 个黑键全部正确显示。

### 2. 录制功能变量声明顺序

**问题：** 录制状态下点击钢琴键，音符没有被记录。虽然按钮显示"停止录制"（说明 `startRecording()` 已执行），但 `recordedNotes.length` 始终为 0。

**解决：** 排查后发现录制状态变量 `isRecording` 和音符缓存 `allNotesCache` 声明在 `triggerNote()` 函数之后。虽然 JavaScript 有 hoisting 机制，但在执行环境中存在时序敏感问题。将全部录制变量和 `allNotesCache` 移至 `triggerNote` 之前声明后修复。

### 3. 音频转谱 — 自相关音高检测

**问题：** 需要从系统音频流中实时识别出单个音符，转为 MIDI 音符号记录到钢琴。

**解决：** 实现了一个基于 autocorrelation（自相关）的 pitch detection 算法：每 120ms 从 AnalyserNode 获取时域数据 → 计算自相关函数找最佳周期 → 周期转频率 → 频率转 MIDI 音符号。当音符变化时记录到录制列表。该方案对单音旋律效果较好，和弦等复杂信号存在局限。

### 4. 钢琴键全宽自适应

**问题：** 4 个八度共 28 个白键，原来用 `max-width: 700px` 限制宽度并加横向滚动条，用户向右滑时看不到右侧键盘。

**解决：** 移除 `max-width` 和滚动包装器，改用 `flex: 1 1 0` 使所有白键自动平分屏幕宽度。黑键使用 `clamp(16px, 3.3cqw, 26px)` 自适应大小。`container-type: inline-size` + `cqw` 单位确保响应式。

### 5. QWERTY 键盘映射

**问题：** 原有映射使用 Z 行 + Q 行 + 数字键，映射逻辑不够直观，用户希望从左到右按 QWERTY 布局顺序对应钢琴键。

**解决：** 重新设计了 28 键映射：QWERTYUIOP → C2–E3，ASDFGHJKL → F3–G4，ZXCVBNM → A4–G5，;' → A5–B5。键盘提示条同步更新，← → 移调保留。

## 五、对 AI 辅助编程的心得体会

这次实训让我深刻体会到 AI 辅助编程的价值：

1. **加速原型开发：** 从零开始构建一个完整的音乐应用，传统方式需要大量查阅 Web Audio API 文档，AI 可以快速生成可运行的代码骨架。

2. **降低调试成本：** 录制 Bug 的排查过程耗时较长，但 AI 能快速分析代码逻辑，指出可能的变量声明顺序问题。虽然 AI 不能 100% 准确定位所有问题，但能大幅缩小排查范围。

3. **文档自动生成：** AI 协助生成的 Prompt 日志、API 文档和报告质量很高，减少了文档编写时间。

4. **局限性认识：** AI 在处理精细的视觉布局（如黑键定位）时，需要人工仔细校验生成代码的正确性。AI 生成的值可能看起来合理但实际有偏移，必须有测试验证。
